# 게임 개선 코딩 가이드 제안서

## 문서 목적

이 문서는 현재 `RING DYNASTY` 코드 구조를 기준으로, 사용자가 제안한 "챕터 > 구간 > 연속 스테이지" 구조를 안전하게 도입하기 위한 구현 가이드다.

이번 버전은 다음 제안 내용을 기준으로 작성한다.

- C등급 챕터 `INDIE CIRCUIT`
- B등급 챕터 `REGIONAL CIRCUIT`
- A등급 챕터 `NATIONAL CIRCUIT`
- 구간 내 연속 진행
- 포기 시 구간 처음부터 재도전
- 스테이지 사이 15초 인터미션
- 보스 특수 능력

이 문서는 이후 사용자가 추가 요구를 던질 때마다 계속 업데이트하는 기준 문서로 사용한다.

---

## 0. 전면 개편 선언

이 문서는 더 이상 `기존 서킷/기존 자동전투와 병행`을 전제로 하지 않는다.

이번 개편의 기준은 다음과 같다.

- 스테이지 전투 배치 시스템은 현행 `주간 메인 경기 1회 + 랭킹 상대 1명` 구조를 전면 교체한다.
- 전투 시스템은 현행 `자동 선택 전투`를 플레이어용 메인 시스템으로 유지하지 않는다.
- 사용자가 지금까지 제안한 요소는 축소 적용하지 않는다.
- 일부만 먼저 넣고 나머지를 나중에 붙이는 형태로 출시하지 않는다.
- 중간 타협형 혼합안은 금지한다.

즉 최종 목표는 아래 두 가지를 동시에 달성하는 것이다.

1. 진행 구조를 `챕터 > 구간 > 스테이지 > 인터미션`으로 완전 전환
2. 전투 구조를 `턴마다 3개 선택지 + 15초 제한 + 상태이상 + 보스 패턴 + 소모품`으로 완전 전환

기존 코드는 참고 자산일 뿐이며, 플레이어가 실제로 경험하는 메인 흐름은 새 시스템으로 교체한다.

---

## 0-1. 핵심 플레이 감각

이번 구조의 목표는 단순히 시스템을 많이 넣는 것이 아니라, 모든 요소가 하나의 압박감으로 연결되게 만드는 것이다.

연결 구조:

- 스테이지 클리어
  - 레전드 카드 1장으로 끝까지 밀어붙이기 어렵다.
  - 연속전이라 HP/ST/FIN이 누적 소모된다.
- 매 턴 선택지
  - 스탯이 높아도 판단을 틀리면 그대로 무너질 수 있다.
- 상태이상 시스템
  - 이번 턴 손해가 다음 스테이지 부담으로 이어진다.
- 아이템 관리
  - 지금 쓸지, 보스를 위해 아낄지 계속 고민하게 만든다.
- 선수 특성별 선택지 풀
  - 강한 선수라도 구간 특성과 패턴이 맞지 않으면 불리하다.

즉 이 시스템의 긴장감은 아래 세 가지 판단을 계속 강요하는 데서 나온다.

1. 어떤 선수를 어떤 구간에 보낼 것인가
2. 매 턴 어떤 선택지를 고를 것인가
3. 아이템을 언제 사용할 것인가

문서 전체의 설계 기준은 이 감각을 유지하는 것이다. 개별 시스템이 따로 노는 구조가 아니라, 서로 연결되어 `다키스트 던전식 압박감`을 만드는 구조여야 한다.

## 1. 현재 코드 구조 분석

### 1-1. 현재 서킷/주간 진행 핵심 위치

- `index.html`
  - `createDefaultState()`
  - `createDefaultCircuitProgress()`
  - `sanitizeCircuitProgress()`
  - 저장 데이터의 기본 상태와 세이브 호환을 관리한다.
- `data/circuit_rework.js`
  - `CIRCUIT_LADDER`
  - `getCurrentCircuitOpponent()`
  - `startWeeklyMatches()`
  - `advanceToNextWeek()`
  - `calculatePostMatchCarryCondition()`
  - 현재 서킷 상대 정의, 메인 경기 진행, 주간 결산, 컨디션 이월 계산이 여기에 모여 있다.
- `data/management_map.js`
  - 메인 경기 1회 + 사이드 경기 최대 2회 중심으로 운영 화면을 구성한다.
  - 현재 UI는 "이번 주 경기 편성" 단위로 설계되어 있다.
- `data/ppv_invaders.js`
  - PPV 보스형 적과 특수 능력 처리의 선례가 이미 있다.
  - 향후 챕터 보스 특수 패턴 시스템은 이 파일의 설계를 참고하는 것이 가장 현실적이다.

### 1-2. 현재 구조의 성격

현재 게임은 기본적으로 아래 흐름이다.

1. 한 주에 메인 경기 1회 진행
2. 필요하면 사이드 경기 추가 진행
3. 주간 결산
4. 다음 주로 넘어가며 컨디션 회복

즉, 현재 구조는 `주 단위 운영 시뮬레이션`에 최적화되어 있고, 사용자가 제안한 `한 번의 도전 안에서 여러 스테이지를 연속으로 통과하는 런(run)` 구조는 아직 없다.

### 1-3. 현재 구조와 신규 제안의 충돌 지점

사용자 제안은 다음 성격을 가진다.

- 한 구간 안에서 여러 스테이지를 끊김 없이 연속 진행
- 이전 스테이지의 HP/ST/FIN을 유지
- 구간 중간 포기 가능
- 포기/패배 시 해당 구간 처음부터 재시작
- 보스마다 특수 규칙 부여

반면 현재 구현은 다음 전제를 깔고 있다.

- 메인 경기는 주간 기준으로 1회
- 경기 종료 후 결산이 즉시 발생
- 다음 주에 회복량이 일괄 적용
- 상대는 랭킹 기반 1명씩 순차 등장

따라서 신규 시스템은 기존 `circuitProgress`를 바로 변형하기보다, 별도 런 상태를 추가하는 방식이 안전하다.

## 2. 구현 방향 재정의

### 2-1. 교체 대상

이번 개편에서 실제로 교체해야 하는 대상은 다음과 같다.

- `CIRCUIT_LADDER` 중심 메인 진행 구조
- `currentRank` 중심 서킷 progression
- `weeklySchedule.mainMatch` 중심 메인 경기 배치
- `startWeeklyMatches()`의 메인 경기 시작 경로
- `runBattleSimulation()` 기반 자동 플레이어 전투 경로
- `cards` 탭의 메인 경기 편성 UI

즉 `메인 진행`과 `메인 전투`는 기존 것을 유지하지 않는다.

### 2-2. 남겨둘 것과 폐기할 것

남겨둘 수 있는 것:

- 레슬러 데이터
- 스탯 계산 자산
- 경기 애니메이션용 캔버스 자산
- 경제/주간 정산의 일부 유틸
- 저장/불러오기 뼈대

폐기 또는 은퇴 대상:

- 랭킹형 메인 서킷 UI
- 플레이어 입력 없는 메인 전투 흐름
- 메인 상대를 주간 단위로 하나씩 배치하는 구조

중요:

- `자동전투`는 개발용 테스트나 시뮬레이션 검증용으로 남길 수는 있다.
- 하지만 플레이어용 메인 경험으로는 사용하지 않는다.

### 2-3. 최종 방향

최종 방향은 `기존 위에 신규 모드 추가`가 아니라 아래 구조다.

```text
기존 메인 서킷/자동전투
-> 개발 참고 자산으로 축소

새 메인 진행
-> 챕터 선택
-> 구간 진입
-> 스테이지 전투
-> 인터미션
-> 다음 스테이지
-> 구간 클리어
-> 챕터 해금
```

즉 이번 개편은 기능 추가가 아니라 `메인 루프 교체`다.

## 3. 새로 추가할 데이터 계층 제안

### 3-1. 신규 데이터 파일

권장 신규 파일:

- `data/chapter_stage_data.js`

역할:

- 챕터
- 구간
- 스테이지
- 보상
- 인터미션 설정
- 보스 특수 규칙

을 선언형 데이터로 관리한다.

### 3-2. 데이터 설계 원칙

- UI 텍스트와 게임 규칙 데이터를 분리한다.
- "보스 여부"는 플래그가 아니라 `boss` 객체 존재 여부로 판단한다.
- 스테이지 수는 구간마다 가변 길이 배열로 처리한다.
- 보상은 각 스테이지 단위와 구간 클리어 단위를 분리한다.
- 경기 강제 규칙은 `modifiers` 배열로 선언한다.

`unlock` 조건은 아래 타입만 사용한다.

- `{ type: "default" }`
- `{ type: "clear_chapter", id: "chapter_c" }`
- `{ type: "clear_segment", id: "c_3" }`
- `{ type: "player_level", value: 10 }`

즉 챕터 해금 규칙도 선언형 데이터로 고정한다.

### 3-4. C/B/A 챕터 반영 원칙

이번 문서에서는 C/B/A 챕터를 모두 메인 progression에 포함하는 것을 전제로 한다.

적용 기준:

- C 챕터: 사용자 제안 수치/보상/보스 규칙을 직접 반영
- B 챕터: 사용자 제안의 구간 수, 스테이지 수, 적 스탯 범위, 보스 이름/특징을 직접 반영
- A 챕터: `B-3 이후 해금`, `동일한 3/4/5 스테이지 구조`, `상위 난이도 확장`을 고정 규칙으로 반영

챕터 해금 조건은 아래처럼 고정한다.

- `chapter_c`
  - `unlock: { type: "default" }`
- `chapter_b`
  - `unlock: { type: "clear_chapter", id: "chapter_c" }`
- `chapter_a`
  - `unlock: { type: "clear_segment", id: "b_3" }`

즉 A 챕터는 "나중에 생각"이 아니라, 지금부터 구조상 반드시 존재해야 하는 챕터다.

아직 이름이나 세부 수치가 완전히 주어지지 않은 구간은 아래 방식으로 정의한다.

- 구조는 지금 확정
- 해금 규칙은 지금 확정
- 데이터 슬롯은 지금 생성
- 미제공 텍스트/수치는 `TODO`가 아니라 `규칙 기반 생성`으로 채움

이 문서 기준에서는 `챕터 구조 미반영 상태로 출시`를 허용하지 않는다.

## 4. 저장 상태 설계 제안

### 4-1. 신규 상태 키

`gameState`는 메인 진행이 교체되므로 아래처럼 재설계하는 것이 맞다.

```js
chapterProgress: {
  unlockedChapterIds: ["chapter_c"],
  clearedSegmentIds: [],
  clearedChapterIds: [],
  lastSelectedChapterId: "chapter_c",
  lastSelectedSegmentId: "c_1"
},
chapterRunState: {
  active: false,
  chapterId: "",
  segmentId: "",
  stageIndex: 0,
  selectedWrestlerId: "",
  runStartedWeek: 0,
  currentHpRatio: 1,
  currentStamina: 100,
  currentFinisherGauge: 0,
  activeStatuses: [],
  actionHistory: [],
  clearedStages: [],
  pendingIntermission: false,
  intermissionEndsAt: 0,
  abandoned: false,
  bossFlags: {},
  itemInventory: [],
  rewardsPreview: {
    gold: 0,
    tickets: 0,
    fragments: {}
  }
}
```

### 4-2. 왜 별도 상태가 필요한가

현재 `circuitProgress`는 아래 목적에 최적화되어 있다.

- 현재 랭킹
- 연승/연패
- 복수전 가능 여부
- 최종 상대 기록

여기에 챕터 런 상태를 억지로 넣으면 책임이 섞인다.

- 랭킹형 진행
- 연속 스테이지 진행
- 인터미션 타이머
- 보스 규칙 처리

따라서 메인 진행 기준으로는 아래처럼 정리해야 한다.

- `circuitProgress`: 구세대 메인 서킷 데이터, 마이그레이션용
- `chapterProgress`: 새 메인 progression 데이터
- `chapterRunState`: 현재 구간 진행 중 런타임 데이터

즉 메인 경험은 `chapterProgress + chapterRunState`가 담당한다.

### 4-3. 저장 호환 처리

수정 위치:

- `index.html`의 `createDefaultState()`
- `index.html`의 `sanitizeState()`

원칙:

- 세이브 파일에 `chapterProgress`, `chapterRunState`, `consumableInventory`가 없으면 기본값 주입
- 런 도중 저장된 경우에도 재접속 가능해야 함
- 단, 인터미션 타이머는 실제 시간 기반보다 "재접속 시 즉시 복귀 가능한 대기 상태"가 안전함

권장 처리:

- 저장/로드 후에는 `pendingIntermission = true`를 유지하고 즉시 인터미션 HUD를 복원한다.
- 기존 `circuitProgress.currentRank`는 로드 시 읽을 수는 있지만, 메인 경로에서는 더 이상 사용하지 않는다.

## 5. 전투 흐름 설계 제안

### 5-1. 기본 런 흐름

```text
챕터 선택
-> 구간 진입
-> 스테이지 1 전투
-> 승리 시 현재 HP/ST/FIN 및 유지 상태 저장
-> 인터미션 진입
-> 스테이지 2 전투
-> ...
-> 보스 전투
-> 구간 클리어 보상 지급
-> 다음 구간 해금
```

### 5-2. 패배/포기 처리

규칙은 사용자 제안을 그대로 따른다.

- 구간 중 패배: 해당 구간 진행도 초기화
- 구간 중 포기: 해당 구간 진행도 초기화
- 이미 클리어한 이전 구간은 유지

즉, 저장 기준은 `챕터 전체`가 아니라 `구간 클리어 단위`다.

### 5-2-1. 구간 시작 시 배치 규칙

이번 개편에서 `배치`는 현행 주간 카드 편성이 아니라 `구간 도전자 선택`을 의미한다.

고정 규칙:

- 구간 시작 전 플레이어는 도전자 1명을 선택한다.
- 선택된 선수는 해당 구간이 끝날 때까지 교체할 수 없다.
- 인터미션에서는 아이템 사용과 전략 변경만 가능하고 선수 교체는 불가하다.
- 다른 선수로 다시 도전하려면 해당 구간을 포기하고 처음부터 재진입해야 한다.

즉 배치 전략의 핵심은 `누구를 이번 구간 러너로 세울 것인가`다.

### 5-2-2. 주간 도전권 규칙

챕터/구간 도전은 무제한이 아니라 `주간 도전권`으로 제한한다.

기본 규칙:

- 기본 도전권: `주당 3회`
- 구간 도전 시작 시 `-1회`
- 중간 포기 시에도 소모 유지
- 구간 클리어 시에도 소모 유지

추가 획득 규칙:

- 시설 업그레이드: `주당 +1회`
- 미션 보상: `+1회`, 일회성
- PPV 클리어 후 다음 주: `+2회`

의도:

- 무한 재시도로 난이도를 무력화하지 못하게 함
- 어떤 구간에 이번 주 도전권을 쓸지 판단하게 함
- PPV 종료 후 다음 주의 러시 타이밍을 강화함

권장 저장 필드:

```js
chapterAttemptState: {
  weeklyMax: 3,
  weeklyBonus: 0,
  weeklyUsed: 0,
  missionBonusCharges: 0,
  ppvClearBonus: 2,
  lastResetWeek: 1
}
```

### 5-3. HP/ST/FIN 이월 방식

현재 `data/circuit_rework.js`에는 `calculatePostMatchCarryCondition()`이 이미 있다. 하지만 새 시스템의 핵심 자원은 `컨디션`이 아니라 `HP / 스태미나 / 피니셔 게이지`다.

이유:

- 현재 함수는 주간 운영용 장기 피로에 가깝다.
- 새 챕터 런은 전투 간 직접 이월 자원이 완전히 다르다.

권장 신규 함수:

- `buildStageCarryState()`

권장 규칙:

- `hpRatio`는 전투 종료 상태 그대로 사용
- `currentStamina`는 전투 종료 시점 그대로 이월
- `finisherGauge`는 전투 종료 시점 그대로 이월
- 허용된 상태이상만 이월

예시:

```js
function buildStageCarryState(battleState) {
  return {
    hpRatio: clamp(battleState.currentHP / battleState.maxHP, 0.01, 1),
    currentStamina: clamp(Math.round(battleState.currentStamina), 0, 100),
    finisherGauge: clamp(Math.round(battleState.finisherGauge), 0, 100),
    activeStatuses: serializeCarryStatuses(battleState.statusEffects)
  };
}
```

의도:

- 이전 스테이지의 선택이 다음 스테이지 전략에 직접 영향을 줌
- 전투 성과와 리스크가 그대로 누적됨

### 5-4. 인터미션 설계

인터미션은 `15초`로 고정한다.

허용할 행동:

- 회복 아이템 사용
- 전술 변경
- 다음 스테이지 정보 확인
- 포기

구현 방식:

- 실제 15초 자동 카운트다운
- 수동 `다음 스테이지` 버튼 제공
- 타이머 종료 시 자동 다음 스테이지 진입
- 인터미션은 축소 불가 시스템이며, 아이템 운영의 핵심 레이어로 본다

## 6. 보스 특수 능력 설계

### 6-1. 기존 선례 활용

`data/ppv_invaders.js`에는 이미 다음 개념이 있다.

- 특정 적 ID별 규칙 분기
- 전투 요약에 능력 문구 추가
- 보상 차등 처리

챕터 보스도 같은 패턴을 재사용하는 것이 좋다.

### 6-2. 구현 원칙

보스 능력은 코드 하드코딩보다 `modifier` 기반 선언형 처리로 가야 한다.

권장 타입:

- `forced_match_type`
- `opening_guard`
- `finisher_resist_once`
- `stat_boost_below_hp`
- `regen_per_turn`
- `rage_on_low_hp`

### 6-3. 권장 처리 계층

- 데이터 + 해석기: `data/chapter_stage_data.js`
- 보스 해석 함수: `applyChapterBossModifiers()`
- 전투 주입 지점: 전투 시작 직전 / 턴 진행 중 / 결과 정산 직후

즉 보스 시스템은 별도 파일로 분리하지 않는다.

이유:

- 보스 데이터와 해석기가 같은 파일에 있으면 보스 추가/수정 시 수정 지점이 1곳으로 고정된다.
- 현재 규모에서는 파일 분리보다 응집도가 더 중요하다.
- `ppv_invaders.js`처럼 패턴 데이터와 처리 규칙이 가까이 있는 구조를 따라가는 편이 관리가 쉽다.

### 6-4. 보스 능력 반영 범위

보스는 단순 수치 상승이 아니라 반드시 패턴을 가져야 한다.

최소 반영 범위:

- 경기 타입 강제
- 예고형 패턴
- HP 임계점 페이즈 변화
- 회복/버프/강공격 예약
- 플레이어 대응 전용 선택지 주입

즉 보스전은 일반 적 전투에 숫자만 올린 버전이 아니어야 한다.

## 9. 밸런스 가이드

### 9-1. C 챕터 기준 밸런스 의도

현재 스타터 로스터와 초기 성장 속도를 고려하면 C 챕터는 아래 경험을 줘야 한다.

- C-1: 시스템 학습
- C-2: 연속전 압박 체험
- C-3: 첫 번째 빌드 체크

즉 C 챕터는 "초반 벽"이 아니라 "새 시스템 튜토리얼 + 첫 보스 러시"여야 한다.

### 9-2. 보상 밸런스 가이드

권장 원칙:

- 스테이지 개별 승리 보상은 작게
- 구간 클리어 보상은 크게
- 플레이어의 기대 보상은 "보스 클리어"에 집중

예:

- 개별 스테이지 골드: 낮게
- 구간 클리어: 티켓/조각 포함
- 챕터 최종 구간: 다음 등급 준비용 핵심 보상 제공

### 9-3. B급 확정 조각 보상 의미

`C-3` 클리어의 `B급 확정 조각`은 매우 중요하다.

이 보상은 다음 역할을 해야 한다.

- B 챕터 진입 동기 제공
- 플레이어의 첫 중기 목표 제공
- 뽑기 운이 나빠도 성장감 유지

따라서 단순 골드보다 성장 체감이 높은 보상으로 유지하는 것이 맞다.

## 13. 다키스트 던전 스타일 전투 분석

### 13-1. 현재 전투 코드 구조

현재 전투 핵심은 `index.html` 내부의 아래 함수들이다.

- `createBattleState()`
- `runBattleSimulation()`
- `chooseMove()`
- `canUseMove()`
- `calcDamage()`
- `calcAttackSuccessChance()`
- `calcFinisherSuccessChance()`
- `resolveSubmission()`
- `drawBattleEventFrame()`

현재 구조의 성격은 다음과 같다.

- 내부적으로는 `턴 단위`로 전투를 계산한다.
- 그러나 플레이어 입력은 없다.
- 매 턴 행동은 `chooseMove()`가 자동 선택한다.
- 전투 후에는 `battleEvents` 타임라인만 애니메이션으로 재생한다.

즉, 이미 턴 기반 계산 자산은 존재하지만, 플레이어용 메인 전투 경험은 사실상 새 엔진으로 다시 짜야 한다.

### 13-2. 현재 전투 상태에서 재사용 가능한 요소

이미 재사용 가능한 자산:

- HP
- 스태미나
- 피니셔 게이지
- 다운 상태
- 핀폴/서브미션 판정
- 모멘텀
- 하이라이트 기록
- 턴 수 제한

즉 신규 시스템은 아래를 새로 만들면 된다.

- 플레이어 행동 선택 UI
- 선택지 생성기
- 15초 타이머
- 행동별 세부 규칙
- AI 적 행동 선택기 고도화
- 그래플/잡기 상태 전이

### 13-3. 현재 구조와 신규 제안의 차이

현재 전투:

- 기술 종류가 `basic`, `power`, `technical`, `submission`, `finisher` 정도로 단순
- AI가 자동 선택
- 전투 중 인터랙션 없음

신규 제안:

- 턴마다 3개 선택지 제시
- 15초 안에 반드시 선택
- 조건부 등장
- 위험/보상 구조가 명확한 선택지
- 잡기 상태, 서브미션 유지, 도발, 버티기 같은 전술 행동 추가

따라서 이 변경은 단순 밸런스 조정이 아니라 `전투 의사결정 레이어 추가`다.

## 14. 전투 개편 방향

### 14-1. 전투 엔진 교체 원칙

플레이어용 전투는 기존 `runBattleSimulation()`을 메인 엔진으로 쓰지 않는다.

대신 아래 구조로 교체한다.

- `createBattleSession()`으로 세션 생성
- 턴 시작 효과 처리
- 선택지 3개 생성
- 15초 내 선택 또는 자동 기본 공격
- 행동 결과 계산
- 상태이상/패턴/소모품 처리
- 로그 및 연출 누적
- 승패 판정

즉 기존 자동전투 함수는 참고용 수학 자산만 흡수하고, 메인 전투 흐름은 새 세션 엔진이 담당한다.

### 14-2. 기존 코드의 활용 범위

재사용 가능한 것:

- HP/스태미나/피니셔 계산 아이디어
- 데미지 계산 일부 계수
- 애니메이션 프레임 자산
- 결과 요약 포맷의 일부

재사용하지 않을 것:

- 플레이어 행동 자동 선택
- while 루프 한 번에 완결되는 전투 구조
- 메인 경기용 자동 진행 UX

### 14-3. 최종 원칙

- 플레이어는 매 턴 반드시 판단해야 한다.
- 3개 선택지는 매 턴 랜덤이되 규칙 기반이어야 한다.
- 일반 스테이지와 보스 스테이지 모두 같은 선택형 전투 규칙을 사용한다.
- 보스전은 그 위에 패턴과 반응 선택지가 추가된다.

## 15. 신규 전투 상태 설계 제안

### 15-1. 전투 중 입력 상태

권장 신규 상태:

```js
battleInputState: {
  active: false,
  battleId: "",
  turn: 0,
  actorId: "",
  choiceDeadlineAt: 0,
  remainingMs: 15000,
  choices: [],
  selectedChoiceId: "",
  autoSelected: false
}
```

역할:

- 현재 누구 턴인지
- 선택지 3개가 무엇인지
- 제한 시간이 얼마나 남았는지
- 자동 선택으로 넘어갔는지

를 UI와 엔진 사이에서 연결한다.

### 15-2. 전투 상태 확장

`createBattleState()`에 아래 필드를 추가하는 것을 권장한다.

```js
statusEffects: {
  attackBuffTurns: 0,
  defenseBreakTurns: 0,
  tauntHitPenaltyTurns: 0,
  guardTurns: 0,
  exposedTurns: 0,
  counterTurns: 0,
  stunImmuneTurns: 0
},
grappleState: {
  active: false,
  controllerId: "",
  targetId: "",
  duration: 0
},
submissionState: {
  active: false,
  controllerId: "",
  targetId: "",
  progressTurns: 0
},
lastActionId: "",
bonusDamageNextTurn: 0
```

기존 구조에는 `isGrapped` 오타 기반 단일 상태가 있으나, 이번 제안 수준을 소화하기엔 부족하다. 단일 불리언이 아니라 `잡은 사람/잡힌 사람/지속 턴`이 들어가는 객체형 상태로 바꾸는 것이 맞다.

### 15-3. 왜 객체형 상태가 필요한가

신규 선택지는 대부분 "이번 턴만 끝나는 기술"이 아니다.

- 방어력 감소 2턴
- 다음 턴 데미지 증가
- 상대 행동 취소
- 그래플 후 후속 선택지 분기
- 서브미션 3턴 유지

따라서 이번 개편의 본질은 기술 추가가 아니라 `상태 이상 및 자세 상태 시스템` 도입이다.

## 16. 행동 선택지 시스템 설계

### 16-1. 권장 행동 데이터 파일

신규 파일 권장:

- `data/battle_action_data.js`

역할:

- 행동 ID
- 분류
- 등장 조건
- 비용
- 성공률 공식
- 효과
- 실패 패널티

를 선언형으로 관리한다.

### 16-2. 권장 행동 정의 구조

```js
const BATTLE_ACTION_DEFS = {
  quick_strike: {
    id: "quick_strike",
    category: "attack",
    label: "빠른 공격",
    staminaCost: 5,
    alwaysAvailable: true,
    successFormula: "fixed",
    successValue: 85,
    damageFormula: "power",
    damageMultiplier: 0.7
  },
  heavy_strike: {
    id: "heavy_strike",
    category: "attack",
    label: "강한 공격",
    staminaCost: 18,
    conditions: [{ type: "stamina_gte", value: 20 }],
    successFormula: "fixed",
    successValue: 60,
    damageFormula: "power",
    damageMultiplier: 1.5,
    onHit: [{ type: "target_stamina_delta", value: -10 }],
    onFail: [{ type: "counter_risk_up", value: 30 }]
  }
};
```

### 16-3. 선택지 생성 규칙

핵심 함수:

- `getAvailableBattleActions(actor, defender, context)`
- `rollBattleChoices(actor, defender, context, count = 3)`

권장 규칙:

1. 항상 등장 가능한 기본 행동 풀 확보
2. 조건 충족 행동 추가
3. 현재 상태에 맞는 특수 행동 가중치 상승
4. 중복 카테고리 과다 방지
5. 최종 3개 제시

### 16-4. 선택지 생성 우선순위

권장 우선순위:

1. 항상 행동 가능한 기본기 최소 1개 보장
2. 방어/회복 선택지 최소 1개 보장
3. 상황 특수 선택지 0~1개 가중 등장

이유:

- 3개 모두 공격기로 뜨면 전략성이 사라진다.
- 3개 모두 회복기로 뜨면 템포가 무너진다.
- 항상 기본기 하나는 있어야 타임아웃 자동 행동과도 일관성이 맞는다.

### 16-5. 추천 카테고리 구조

- `core_attack`
- `core_defense`
- `setup`
- `high_risk`
- `grapple_followup`
- `finisher_only`

이 카테고리를 두면 제시 로직과 밸런스 조정이 쉬워진다.

### 16-6. 선수 타입별 선택지 풀 편향

선수 선택은 단순 스탯 비교가 아니라 `어떤 행동 풀이 열리느냐`를 결정해야 한다.

핵심 원칙:

- 어떤 선수도 모든 구간에서 최강이 아니어야 한다.
- 구간 특성과 강제 패턴에 따라 적합한 선수가 달라져야 한다.
- 선택지 생성은 현재 전투 상태뿐 아니라 선수 타입 가중치도 반영해야 한다.

권장 타입별 편향:

#### 파워형

- 자주 등장: `강한 공격`, `돌진`, `무리수`
- 드물거나 제거: `기술기`, `그래플`, `서브미션 시도`

의도:

- 초반 C구간 압살 능력은 좋음
- 기술전/서브미션 강제 보스에서 불리함

#### 테크니션형

- 자주 등장: `기술기`, `그래플`, `서브미션 시도`
- 드물게 등장: `강한 공격`

의도:

- 패턴 대응과 제어 능력이 좋음
- 순간 폭딜은 파워형보다 낮음

#### 쇼맨형

- 자주 등장: `도발`, `쇼맨십`
- 추가 특성: 피니셔 게이지 충전 가속
- 약점: 순수 데미지 기대값 낮음

의도:

- Hype와 FIN 운영에 강함
- 장기전, 피니셔 운영전에서 강세

권장 구현 방식:

- 각 선수는 `actionWeights`를 가진다.
- 선택지 생성 시 행동 등장 확률에 가중치를 곱한다.
- 단, 현재 조건상 반드시 필요한 방어/회복 선택지는 타입 편향보다 우선한다.

예시:

```js
actionWeights: {
  heavy_strike: 1.6,
  rush: 1.4,
 desperation: 1.3,
  technical_move: 0.3,
  grapple: 0.1,
  submission_attempt: 0.0
}
```

즉 `선수 선택 -> 전투 선택지 풀 변화 -> 구간 상성 발생`이라는 구조를 메인 전략으로 삼는다.

## 17. 제안된 행동별 구현 메모

### 17-1. 필수 행동 세트

이번 개편에서 전투 행동은 아래 전체 세트를 모두 포함해야 한다.

- `빠른 공격`
- `강한 공격`
- `기술기`
- `피니셔`
- `도발`
- `돌진`
- `버텨내기`
- `숨 고르기`
- `체력 집중`
- `무리수`
- `쇼맨십`
- `그래플`
- `서브미션 시도`
- `던지기`
- `무릎공격`

즉 행동 풀 일부만 적용한 상태는 완료로 보지 않는다.

### 17-2. 각 행동의 처리 기준

#### 빠른 공격

- 기존 `basic`을 대체 또는 별칭 처리
- 타임아웃 기본 행동으로도 사용
- 기본 행동이므로 항상 등장 가능

#### 강한 공격

- 기존 `power`를 대체하는 핵심 액션
- 실패 시 반격 리스크를 별도 상태값으로 보관해야 함

권장 필드:

- `pendingCounterBonusChance`

#### 기술기

- 기존 `technical` 확장판으로 보는 것이 좋다.
- 명중률은 고정값보다 `기술 스탯 기반 공식`으로 계산
- 방어 감소 디버프는 `defenseBreakTurns = 2`로 처리

#### 피니셔

현재 코드에는 `finisherGauge >= 100`만 확인하면 되지만, 신규 제안은 아래 조건이 추가된다.

- 게이지 100%
- 상대 HP 35% 이하

따라서 `canUseMove()`를 확장해야 한다.

실패 패널티:

- 고정 카운터 데미지 50
- 게이지 -50

이 부분은 현재 코드에 없는 강한 리스크이므로 별도 처리 분기가 필요하다.

#### 도발

기존 전투 엔진에는 순수 버프형 행동이 거의 없다. 따라서 도발은 `non_damage_action` 타입을 새로 도입하는 것이 좋다.

효과:

- Hype +15
- 피니셔 게이지 +20
- 상대 다음 공격 성공률 -20%

위험:

- 상대 파워 80 이상이면 역효과

즉 `context-sensitive action`으로 구현해야 한다.

#### 돌진

기능:

- 공격 + 확률 스턴

권장 처리:

- 성공 시 `defender.isStunned = true`
- 실패 시 `attacker.isDown = true` 확률 30%

#### 버텨내기

핵심 방어기다.

효과:

- 스태미나 +20
- 받는 데미지 50% 감소
- 반격 확률 +40%

현재 엔진에는 "이번 턴에만 적용되는 방어 태세"가 없으므로 다음과 같은 임시 상태가 필요하다.

- `guardTurns = 1`
- `counterTurns = 1`

#### 숨 고르기

효과:

- 스태미나 +35
- 최대 HP의 5% 회복
- 대신 이번 턴 받는 데미지 +30%

권장 상태:

- `exposedTurns = 1`

#### 체력 집중

효과:

- 피니셔 게이지 +30
- 다음 턴 공격 데미지 +20%

권장 상태:

- `bonusDamageNextTurn = 0.2`

## 18. 그래플/서브미션 상태 설계

### 18-1. 구현 포인트

사용자 제안에서 가장 구현 난도가 높은 부분은 그래플 후 후속 선택지 분기다.

필요 상태:

- 누가 잡고 있는가
- 언제까지 유지되는가
- 다음 턴에 어떤 전용 선택지가 열리는가
- 탈출 판정은 어떻게 하는가

### 18-2. 권장 흐름

```text
그래플 성공
-> grappleState.active = true
-> 다음 내 턴에 grapple_followup 선택지 풀 활성화
-> [던지기] [서브미션] [무릎공격] 등장
-> 상대가 먼저 턴을 얻으면 탈출 판정
-> 탈출 성공 시 grappleState 해제
```

### 18-3. 서브미션 유지 규칙

권장 상태:

```js
submissionState: {
  active: true,
  controllerId: "left",
  targetId: "right",
  progressTurns: 1
}
```

매 턴 처리:

- 상대 스태미나 -15
- `progressTurns += 1`
- 3턴 유지 시 탭아웃 승리

탈출 확률은 사용자 제안대로 구현하되, 실제 공식은 코드에서 안전하게 clamp 해야 한다.

권장 공식:

```js
escapeChance = clamp(((target.stats.power + target.stats.technique) * 0.6) * 1.2, 5, 95);
```

원문은 `상대 파워/기술 스탯 × 1.2%`라고 되어 있는데 해석 여지가 있으므로 실제 코딩 단계에서 공식 정의를 문서에 다시 확정하는 것이 필요하다.

## 19. 15초 타이머 시스템 제안

### 19-1. 권장 동작

플레이어 턴 시작:

1. 선택지 3개 표시
2. 15초 카운트다운 시작
3. 시간 내 클릭 없으면 `빠른 공격` 자동 선택
4. 선택 즉시 결과 처리 후 상대 턴 진행

### 19-2. 기술 구현 방식

권장 방식:

- 전투는 `async` 턴 루프로 분리
- 플레이어 턴에서 `await waitForPlayerChoiceOrTimeout()`
- AI 턴은 즉시 `chooseEnemyAction()`

이유:

- 현재 `while` 루프 기반 `runBattleSimulation()`은 전투를 한 번에 계산한다.
- 15초 실시간 입력을 받으려면, 전투 계산을 `한 턴씩 진행하는 state machine` 구조로 분해해야 한다.

즉 여기서 중요한 결론은 하나다.

- 플레이어용 메인 전투는 세션 기반 상태기계로 다시 짠다
- 기존 자동전투 함수는 메인 경로에서 제외한다

권장 신규 함수:

- `createBattleSession()`
- `stepBattleTurn(session, chosenActionId)`
- `resolveAiTurn(session)`
- `finalizeBattleSession(session)`

### 19-3. 자동 기본 공격 처리

타임아웃 시 행동은 사용자 제안대로 `빠른 공격`이 적절하다.

이유:

- 항상 사용 가능
- 리스크가 낮음
- 플레이어 무응답 시 전투가 완전히 꼬이지 않음

### 19-4. 타이머 시각 규칙

카운트다운 표시는 아래처럼 고정한다.

- `15~10초`: 흰색
- `9~5초`: 노란색
- `4~1초`: 빨간색 + 깜빡임
- `0초`: 첫 번째 선택지 자동 실행

의도:

- 남은 시간의 위험도를 색만 보고 즉시 판단 가능하게 함
- 완벽한 판단보다 빠른 판단을 강제하는 긴장감을 유지

### 19-5. 보스 경고 턴 제한 시간

보스 특수 패턴 경고 턴은 일반 턴과 다르게 `10초`로 줄인다.

규칙:

- 일반 턴: `15초`
- 보스 예고/대응 턴: `10초`

이 축소는 보스전 긴장감을 만드는 핵심 장치로 본다.

## 20. UI 설계 제안

### 20-1. 현재 UI와의 관계

현재 전투 애니메이션은 전투가 끝난 뒤 `battleEvents`를 재생하는 방식이다. 다키스트 던전 스타일을 제대로 구현하려면 `실시간 전투 HUD`가 새로 필요하다.

필수 UI:

- 현재 턴 수 `TURN X / LIMIT`
- 남은 시간 `15초`
- 행동 카드 3개
- 내 HP / 스태미나 / 피니셔 게이지
- 상대 HP / 스태미나 / 피니셔 게이지
- 현재 상태 이상 표시
- 그래플/서브미션 상태 표시

### 20-1-1. 전투 화면 기본 레이아웃

전투 HUD는 아래 3단 구조로 고정한다.

1. 상단 상태 바
2. 중앙 전투 캔버스
3. 하단 선택 카드 영역

상단 상태 바에 반드시 포함할 정보:

- 내 선수 정보
- 상대 정보
- `TURN X / LIMIT`
- `STAGE X / TOTAL`
- 현재 전투 상태 문구
- 상대 패턴 상태 예고

상단 자원 표시는 아래 순서로 고정한다.

- `HP`
- `ST`
- `FIN`

중앙 전투 캔버스에는 아래를 표시한다.

- 좌측 내 선수 도트
- 우측 상대 도트
- 중앙 상황 문구

상황 문구 예:

- `정적이 흐른다... Jack이 노려보고 있다`
- `⚠️ Iron Rex가 뭔가를 준비하고 있다`
- `⏱ The Phantom 회복까지 2턴`

하단 선택 영역에는 아래를 표시한다.

- `선택하세요 (남은 초)`
- 선택지 카드 3개
- 전투 중 사용 가능한 아이템 버튼
- `스킵/자동` 버튼

### 20-2. 권장 렌더링 위치

단기적으로는 기존 캔버스 위에 HTML 오버레이를 얹는 것이 가장 현실적이다.

권장 구조:

- 캔버스: 캐릭터/이펙트/링 연출
- HTML 오버레이: 행동 카드, 타이머, 텍스트 정보

이유:

- 행동 카드 3개를 캔버스로만 구현하면 이벤트 처리와 반응형 UI가 불편하다.
- HTML 버튼 + CSS 타이머가 훨씬 유지보수에 유리하다.

### 20-3. 행동 카드 표시 규칙

각 카드에 최소한 아래를 표기해야 한다.

- 행동명
- 예상 효과
- 스태미나 변화
- 성공률
- 리스크 한 줄
- 조건 충족 여부

조건 미충족 카드는 아예 후보 풀에서 제거하는 것이 맞고, 비활성 카드로 보여주는 방식은 현재 단계에서는 비추천이다.

### 20-4. 선택지 카드 시각 규격

각 카드 구조는 아래 순서로 고정한다.

- 상단: 행동 아이콘 + 행동명
- 중단: 예상 핵심 수치
- 하단: 비용 + 성공률 + 부가 효과

카드 색상 규칙:

- 공격 계열: 붉은 계열 배경
- 방어/회복 계열: 파란 계열 배경
- 특수 계열: 보라 계열 배경
- 위험한 선택: 주황 테두리 경고

성공률 시각 규칙:

- `85% 이상`: 초록 점 + `안정`
- `60~84%`: 노랑 점 + `보통`
- `40~59%`: 주황 점 + `위험`
- `39% 이하`: 빨강 점 + `도박`

호버 또는 길게 누르기 툴팁에는 아래를 표시한다.

- 기본 데미지
- 치명타 기대치
- 실패 시 결과
- 반격/패널티
- 상태이상 유발 여부

즉 카드 본체는 빠른 판단용, 툴팁은 정밀 판단용으로 역할을 나눈다.

## 21. 기존 코드에 반영할 실제 작업 순서

- `index.html`

즉 `일반 스테이지는 자동, 보스만 수동` 같은 혼합 운영은 하지 않는다.

## 22. 밸런스 및 UX 주의사항

### 22-1. 가장 큰 리스크

15초 제한과 3지선다 구조는 재미를 만들 수 있지만, 모든 경기에서 강제하면 피로도가 매우 높아질 수 있다.

이번 개편에서는 이 부담을 자동전투로 피하는 대신 아래 방식으로 해결해야 한다.

- 전투 연출 길이 단축
- 카드 정보 가독성 강화
- 타임아웃 시 기본 공격 자동 선택
- 인터미션에서 자원 정리 가능
- 보스 패턴 예고를 명확히 표시

### 22-2. 선택지 RNG 리스크

선택지가 완전 랜덤이면 억까처럼 느껴질 수 있다.

필수 규칙:

- 항상 기본 공격 1개 보장
- 항상 방어/회복 계열 1개 보장
- 피니셔 조건 충족 시 높은 확률로 피니셔 등장
- 그래플 후에는 후속 선택지가 우선 등장

### 22-3. 출시 승인 기준

이번 개편은 아래가 모두 들어가야 출시 승인 가능하다.

- 챕터/구간/스테이지 progression
- 모든 스테이지의 선택형 전투
- 행동 풀 전체
- 상태이상 전체
- 보스 패턴
- 인터미션
- 소모품 구매/보유/사용
- HP/ST/FIN 이월

하나라도 빠지면 시스템 정체성이 무너진다.

## 23. 전투 중 상태이상 설계

### 23-1. 현재 코드 기준 주의점

현재 프로젝트에서 `부상`은 이미 경기 외 장기 결장 상태 의미로 사용된다.

예:

- 로스터 상태 `injured`
- `injuryWeeks`
- 치료 슬롯
- 주간 회복/예방 시스템

따라서 전투 중 상태이상으로서의 "부상(Bleeding)"은 같은 이름을 그대로 쓰면 안 된다.

권장 네이밍:

- 장기 부상: `injured`, `injuryWeeks`
- 전투 출혈: `bleeding`
- 전투 탈진: `exhausted`
- 전투 공포: `rattled`
- 전투 기절: `stunned`
- 패턴 적응: `patternRead` 또는 `repetitionPenalty`

### 23-2. 권장 상태이상 데이터 구조

`battleState.statusEffects`를 아래처럼 확장하는 것을 권장한다.

```js
statusEffects: {
  bleeding: { turns: 0, hpPercentPerTurn: 0 },
  exhausted: { active: false },
  stunned: { turns: 0 },
  rattled: { active: false },
  repetitionPenalty: {
    actionId: "",
    stacks: 0,
    active: false
  },
  defenseBreak: { turns: 0, value: 0 },
  guard: { turns: 0, damageReduction: 0 },
  exposed: { turns: 0, damageTakenBonus: 0 },
  counter: { turns: 0, chanceBonus: 0 },
  focus: { turns: 0, accuracyBonus: 0 },
  nextAttackBuff: { turns: 0, damageBonus: 0 }
}
```

### 23-3. 전투 중 상태이상 적용 순서

권장 처리 순서:

1. 턴 시작 상태이상 틱 처리
2. 자동 해제 조건 확인
3. 선택지 생성
4. 행동 실행
5. 행동 후 부가 상태이상 부여
6. 턴 종료

이 순서를 지켜야 "출혈로 쓰러짐", "기절이라 선택 불가", "탈진으로 강한 공격이 안 뜸" 같은 규칙이 일관된다.

### 23-4. 상태이상별 구현 메모

#### 출혈 `bleeding`

발생:

- `강한 공격` 명중 시 15%

효과:

- 매 턴 최대 HP의 3% 감소
- 3턴 지속

해제:

- `숨 고르기`
- `진통제`

권장 처리:

```js
bleeding: { turns: 3, hpPercentPerTurn: 0.03 }
```

주의:

- 최대 HP 비례 고정 피해이므로 방어력/모멘텀 영향을 받지 않게 해야 한다.
- 전투 로그와 아이콘 표시가 반드시 있어야 한다.

#### 탈진 `exhausted`

발생:

- 스태미나 0 도달 시 자동

효과:

- 공격 성공률 -30%
- 받는 데미지 +20%
- `강한 공격`, `기술기` 제거

해제:

- 스태미나 30 이상 회복

권장 처리:

- 상태이상 객체보다 `derived state`로 계산하는 편이 맞다.
- 즉 `currentStamina <= 0`이면 활성, `>= 30`이면 해제

이유:

- 별도 부여/해제 로직보다 현재 스태미나 기준 파생 상태가 더 단순하다.

#### 기절 `stunned`

현재 코드에도 이미 `isStunned`가 있다. 다만 신규 수동전투에서는 "선택 불가" 규칙으로 해석을 바꿔야 한다.

권장 동작:

- 플레이어 턴이면 버튼 선택 비활성
- 자동으로 `빠른 공격` 또는 `헛손질` 처리

문서 원안은 "자동으로 기본 공격"이지만, 기절 상태에서 기본 공격을 하게 하면 체감상 기절이 약하다. 코딩 단계에서 아래 둘 중 하나를 확정할 필요가 있다.

- A안: 자동 기본 공격
- B안: 행동 스킵

현재 제안서 기준 추천은 `A안`이다. 사용자 원문과 더 가깝고 전투 템포도 덜 끊긴다.

#### 공포 `rattled`

발생:

- HP 20% 이하 최초 진입 시

효과:

- 공격 성공률 -15%
- `무리수` 가중 등장 또는 강제 등장

해제:

- 없음

권장 처리:

- 1회 발동 플래그 필요

```js
rattled: { active: false, triggeredAtLowHp: false }
```

#### 패턴 적응 `repetitionPenalty`

발생:

- 같은 행동 3회 연속 사용

효과:

- 해당 행동 성공률 -25%

해제:

- 다른 행동 1회 사용

권장 상태:

```js
actionHistory: ["quick_strike", "quick_strike", "quick_strike"],
repetitionPenalty: {
  actionId: "quick_strike",
  active: true,
  accuracyPenalty: 25
}
```

이건 상태이상이라기보다 `행동 패턴 읽힘` 디버프이므로, 별도 아이콘과 함께 보여줘야 한다.

## 24. 보스 패턴 시스템 확장

### 24-1. 현재 코드 선례와 연결

기존 프로젝트에는 `data/ppv_invaders.js`에서 보스형 적의 특수 능력을 결과 문구와 보상 처리에 반영한 선례가 있다. 이번 보스 패턴은 그보다 한 단계 더 나아가 `턴 예약형 패턴`까지 포함해야 한다.

즉 신규 보스 패턴 시스템은 아래 3가지를 가져야 한다.

- 조건부 패턴 발동
- 다음 턴 예고
- 특정 턴 강제 행동

### 24-2. 권장 보스 패턴 상태

```js
bossPatternState: {
  patternId: "",
  phase: "",
  telegraphActive: false,
  telegraphText: "",
  nextForcedActionId: "",
  recoveryCountdown: 0,
  enraged: false
}
```

### 24-3. Iron Rex 패턴 설계

트리거:

- HP 50% 이하 최초 진입

진행:

1. `분노 모드` 발동
2. 현재 턴 또는 즉시 다음 턴에 예고 텍스트 표시
3. 다음 턴 `강공격 확정`

플레이어 전용 대응 선택지:

- `막기 준비`
- `선공`
- `회피`

권장 구현 방식:

- 일반 행동 풀 외에 `bossReactionChoices`를 추가
- 예고 상태일 때는 일반 선택지 3개를 완전히 대체하지 말고, 반응 선택지 1~2개를 강제 포함

이유:

- 사용자가 대응해야 하는 패턴인데 일반 랜덤 선택지에 묻히면 의미가 없다.

권장 예시:

```js
if (bossPatternState.telegraphActive) {
  injectPriorityChoices(["guard_prepare", "preemptive_strike", "evade_pattern"]);
}
```

### 24-4. The Phantom 패턴 설계

핵심:

- 매 5턴마다 회복
- 남은 턴 카운트 예고

권장 상태:

```js
bossPatternState: {
  recoveryCountdown: 3,
  nextForcedActionId: "phantom_regen"
}
```

권장 표시:

- `3턴 후 회복`
- `2턴 후 회복`
- `1턴 후 회복`

실제 발동:

- HP 10% 회복
- 로그에 회복량 명시

이 패턴은 공격 강제보다 `시간 압박형 보스`로 설계하는 것이 맞다.

### 24-5. 패턴 구현 원칙

- 보스 패턴은 `행동 카드`와 별도 계층으로 관리
- 일반 기술 선택 로직에 직접 하드코딩하지 말 것
- `턴 시작 훅`, `HP 임계점 훅`, `턴 종료 훅`을 기준으로 처리

권장 훅:

- `onBattleStart`
- `onTurnStart`
- `onHpThresholdCrossed`
- `onTurnEnd`

## 24-6. 상대 행동 예고 시스템

상대 행동은 완전 공개하지 않고 `예고 아이콘`으로만 힌트를 준다.

표시 규칙:

- `⚡` 빠른 공격 예정
- `강공격 아이콘` 강한 공격 예정
- `방어 아이콘` 방어/회복 예정
- `❓` 알 수 없음 또는 보스 패턴

정확도 규칙:

- 일반 적: `80%` 확률로 예고대로 행동
- 보스: `60%` 확률로 예고대로 행동

의도:

- 예고는 전략 재료이지 정답지가 아님
- 플레이어가 예고를 믿고 대응할지, 공격적으로 읽을지 판단해야 함

구현 기준:

- `telegraphActionId`
- `telegraphConfidence`
- `telegraphIcon`

를 턴 시작 시 계산하고 HUD에 표시한다.

## 25. 스테이지 클리어 후 인터미션 상세 설계

### 25-1. 현재 구조와 차이

현재 프로젝트는 주간 경기 종료 후 곧바로 `pendingWeeklySummary`로 결산한다. 하지만 챕터 런은 스테이지 클리어 직후 `중간 정산 없는 인터미션`이 필요하다.

따라서 챕터 런 전용 중간 상태를 둬야 한다.

권장 상태:

```js
chapterIntermissionState: {
  active: false,
  chapterId: "",
  segmentId: "",
  clearedStageId: "",
  nextStageId: "",
  selectedWrestlerId: "",
  carryHpRatio: 1,
  carryStaminaRatio: 1,
  carryFinisherGauge: 0,
  remainingSeconds: 15,
  usedItemIds: [],
  notes: []
}
```

### 25-2. 인터미션에서 보여줄 핵심 정보

- 현재 HP
- 현재 스태미나
- 현재 피니셔 게이지
- 다음 상대 이름
- 예상 강도
- 상대 성향
- 보유 아이템
- 사용 가능한 아이템
- 자동 진행까지 남은 시간

### 25-3. 인터미션에서 유지해야 할 자원

사용자 제안의 핵심은 `이전 스테이지를 어떻게 싸웠는지가 다음 스테이지에 직접 영향을 줌`이다.

따라서 아래 값은 그대로 이월되어야 한다.

- HP
- 스태미나
- 피니셔 게이지
- 전투 중 해제되지 않은 일부 상태이상 여부

권장 정책:

- `bleeding`은 인터미션에서 유지 가능
- `stunned`는 전투 종료 시 해제
- `rattled`는 같은 스테이지 전투 안에서만 유지
- `repetitionPenalty`는 스테이지 종료 시 초기화

즉 모든 상태를 넘기면 과하므로, 이월 가능한 상태와 전투 한정 상태를 구분해야 한다.

### 25-4. 인터미션 시간

시간 규칙은 문서 기준으로 아래처럼 고정한다.

- 전투 중 턴 선택 제한: `15초`
- 스테이지 클리어 후 인터미션: `15초`

이 값은 더 이상 후보안이 아니라 구현 고정값으로 본다.

## 26. 소모품 시스템 설계

### 26-1. 현재 코드 구조 분석 결과

현재 상점 구조는 사실상 다음 3개뿐이다.

- `gacha`
- `fragments`
- `facilities`

즉 소모품 구매/보유/사용 시스템은 아직 없다. 이번 제안은 `경제 시스템 확장`과 `전투 시스템 확장`이 동시에 필요하다.

### 26-2. 권장 저장 상태

`gameState`에는 총 3칸 인벤토리 규칙을 고정 적용한다.

```js
consumableInventory: {
  maxSlots: 3,
  items: []
}
```

이유:

- 사용자 제안의 전략 의도와 더 잘 맞는다.
- 어떤 아이템을 들고 갈지 선택이 생긴다.
- 보스전 대비와 중간 스테이지 버티기 사이의 긴장이 생긴다.

### 26-3. 권장 소모품 데이터 파일

신규 파일:

- `data/consumable_data.js`

예시:

```js
const CONSUMABLE_DEFS = {
  energy_drink: {
    id: "energy_drink",
    label: "에너지 드링크",
    price: 300,
    usePhase: ["intermission"],
    maxOwned: 3,
    effects: [
      { type: "heal_hp_percent", value: 0.20 },
      { type: "restore_stamina_full", value: true }
    ]
  },
  painkiller: {
    id: "painkiller",
    label: "진통제",
    price: 200,
    usePhase: ["intermission"],
    maxOwned: 3,
    effects: [
      { type: "clear_status", value: "bleeding" },
      { type: "heal_hp_percent", value: 0.05 }
    ]
  }
};
```

### 26-4. 소모품별 구현 메모

#### 에너지 드링크

- 인터미션 전용
- HP 20% 회복
- 스태미나 전부 회복

이건 챕터 런 핵심 소모품이므로 가장 먼저 구현해야 한다.

#### 진통제

- 인터미션 전용
- 출혈 해제
- HP 5% 회복

출혈 시스템과 같이 들어가야 의미가 있다.

#### 각성제

- 인터미션 전용 권장
- 피니셔 게이지 +50
- 다음 턴 스태미나 회복 없음

주의:

- "다음 턴" 기준이 인터미션 직후 첫 전투 턴인지, 경기 전체 첫 턴인지 문서에 명확히 고정해야 한다.

권장:

- 다음 스테이지의 첫 턴 회복 없음

#### 응급 처치

- 전투 중 1회 사용 가능
- 턴 소모 없음
- HP 15% 즉시 회복

이 아이템은 다른 소모품과 다르게 `전투 중 사용 허용`이므로 별도 입력 슬롯이 필요하다.

권장 제약:

- 전투당 1회
- 같은 턴 행동과 별도로 사용 가능

#### 집중력

- 다음 공격 성공률 +40%
- 1회 사용

권장 처리:

- `statusEffects.focus = { turns: 1, accuracyBonus: 40 }`

### 26-5. 구매처 설계

현재 상점에 새 서브탭을 추가하는 것이 가장 자연스럽다.

권장 추가:

- `shop > consumables`

즉 상점 탭 구성은 다음처럼 확장된다.

- `gacha`
- `fragments`
- `facilities`
- `consumables`

### 26-6. 사용처 설계

소모품 사용 단계:

- 인터미션 전용 아이템
- 전투 중 허용 아이템

권장 함수:

- `canUseConsumable(itemId, phase, context)`
- `applyConsumableEffect(itemId, targetState, context)`

주의:

- 전투 중 사용 아이템은 `battleSession`과 연결
- 인터미션 아이템은 `chapterIntermissionState`와 연결

즉 같은 아이템 시스템이라도 연결 지점이 다르다.

## 26-7. 스테이지 결과 평가 및 경험치

스테이지 클리어 평가는 `남은 HP 비율` 기준으로 계산한다.

평가 규칙:

- `80% 이상`: `S랭크`
- `50~79%`: `A랭크`
- `30~49%`: `B랭크`
- `29% 이하`: `C랭크`

보상 규칙:

- `S랭크`: 골드 `+50%`, 경험치 `+30%`
- `A랭크`: 골드 `+20%`
- `B랭크`: 추가 보너스 없음
- `C랭크`: 추가 보너스 없음, 다음 스테이지도 낮은 HP로 그대로 진입

즉 이 시스템은 단순 승리 여부가 아니라 `얼마나 깔끔하게 이겼는가`를 평가한다.

### 26-8. 구간 전체 클리어 랭크

구간 전체 평가는 스테이지별 결과를 합산해 계산한다.

추가 보상 규칙:

- 전 스테이지 `S`: 레전드 조각 보너스
- 전 스테이지 `A 이상`: 뽑기권 추가

이 규칙의 목적은 아래와 같다.

- 단순 클리어보다 정교한 플레이를 유도
- 강한 선수만이 아니라 플레이 판단력에도 보상을 줌
- 아이템 사용 시점과 리스크 관리의 의미를 강화

권장 저장 필드:

```js
stageResults: [
  { stageId: "c_2_1", clearRank: "S", endHpRatio: 0.83, bonusGoldRate: 0.5, bonusExpRate: 0.3 }
]
```

## 27. 구현 단계 재정리

### 27-1. 구현 순서와 완료 조건

구현은 순서대로 진행할 수 있지만, 완료 판정은 전체 세트 기준으로 한다.

필수 완료 항목:

- 챕터 progression 전체
- 선택형 전투 전체
- 행동 풀 전체
- 상태이상 전체
- 보스 패턴 전체
- 인터미션 전체
- 소모품 전체

### 27-1-1. 사용자 제안 적용 체크리스트

지금까지 제안된 항목은 아래처럼 모두 메인 시스템에 반영되어야 한다.

- C/B/A 챕터 기반 progression
- 구간별 연속 스테이지
- 포기 시 구간 처음부터 재도전
- 스테이지 사이 15초 인터미션
- 턴마다 3개 선택지 제시
- 15초 내 미선택 시 기본 공격 자동 실행
- 행동 풀 전체
- 상태이상 `bleeding`, `exhausted`, `stunned`, `rattled`, `repetitionPenalty`
- 보스 패턴 `Iron Rex`, `The Phantom`
- HP/ST/FIN 이월
- 인터미션 아이템 사용
- 소모품 구매/보유/전투 중 사용 규칙
- 구간 시작 시 1인 도전자 배치와 구간 중 교체 불가

이 체크리스트는 선택 옵션이 아니라 완료 조건이다.

### 27-2. 기존 문서와 연결되는 핵심 변경점

이번 단계 추가로 문서의 핵심 방향은 아래처럼 강화된다.

- 챕터 런은 `HP/ST/FIN` 이월이 핵심
- 전투 선택지는 상태이상과 아이템이 결합되어야 의미가 생김
- 보스전은 단순 수치 상승이 아니라 `예고형 패턴 대응`으로 설계
- 경제 시스템은 소모품 구매 루프까지 포함해 확장 필요
- 메인 progression과 메인 전투는 기존 시스템을 대체해야 함
